3. 일일 응모권 정책 (3장 cap)

순번트리거광고흐름
1보스 처치XCLAIM_TICKET(adWatched: false) → 서버 적립 → TICKET_GRANTED(count=1)
2보스 처치필수PLAY_AD_REWARDED → AD_COMPLETED → CLAIM_TICKET(adWatched: true) → TICKET_GRANTED(count=2)
3보스 처치필수동일. TICKET_GRANTED(count=3)
4+보스 처치—TICKET_REJECTED(reason='cap_reached')

자정 KST 자동 리셋. 게임 클라가 1분 폴링 + visibilitychange 감지.

4. 서버 API (권장)

POST /api/games/def/claim

요청:
{
  "user_id": "...",
  "source": "boss_kill",
  "adWatched": true,
  "stage": 20,
  "game_id": "DEF"
}

응답 200: { "count": 2, "remaining": 1 }
응답 409: { "error": "cap_reached", "count": 3 }

5. DB 스키마 (예시)

CREATE TABLE polyball_minigame_tickets (
  id BIGSERIAL PRIMARY KEY,
  user_id TEXT NOT NULL,
  game_id TEXT NOT NULL,        -- 'DEF', 'BBADA', 'FISH'
  ticket_date DATE NOT NULL,    -- KST 일자
  source TEXT NOT NULL,
  ad_watched BOOLEAN NOT NULL,
  stage INTEGER,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX ix_def_user_date ON polyball_minigame_tickets (user_id, game_id, ticket_date);

6. 보안 체크리스트

  • parent 측 event.origin 화이트리스트 검증 필수
  • adWatched 검증: 광고 SDK 콜백 토큰 같이 받아 서버에서 재확인
  • source는 화이트리스트 (현재 'boss_kill'만)
  • iframe sandbox: allow-scripts allow-same-origin

7. 광고 정책 필독

  • Rewarded 전용. 인터스티셜/자동 광고 절대 금지.
  • 일 최대 노출: 2회 (응모권 2~3번째).
  • 광고 실패 시 응모권 청구 자동 중단.

8. 표준 흐름 (한눈)

READY → SET_PLAYER
보스 처치 (cap=0) → CLAIM_TICKET(adWatched:false) → TICKET_GRANTED(1)
보스 처치 (cap=1) → PLAY_AD_REWARDED → AD_COMPLETED → CLAIM_TICKET(true) → TICKET_GRANTED(2)
보스 처치 (cap=2) → PLAY_AD_REWARDED → AD_COMPLETED → CLAIM_TICKET(true) → TICKET_GRANTED(3)
보스 처치 (cap=3) → CLAIM_TICKET → TICKET_REJECTED(cap_reached)
